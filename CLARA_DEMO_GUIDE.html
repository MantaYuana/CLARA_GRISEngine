BAB 1 — PENDAHULUAN

1.1 Apa Itu CLARA

CLARA (Contract & Legal AI Reasoning Assistant) adalah asisten kecerdasan buatan yang dirancang khusus untuk membantu pelaku usaha, khususnya UMKM, dalam menganalisis aspek hukum kontrak bisnis dan dokumen legal. CLARA bukanlah chatbot AI generik. CLARA dibangun di atas GRIS Engine (Grounded Retrieval Information System Engine), sebuah mesin retrievasi hybrid yang menggabungkan tiga metode pencarian berbeda secara paralel: dense retrieval (pencarian semantik berbasis vektor), BM25 retrieval (pencarian kata kunci tradisional), dan symbolic retrieval (penelusuran knowledge graph).

Ketiga hasil retrievasi ini difusikan menggunakan Reciprocal Rank Fusion (RRF) untuk menghasilkan konteks hukum yang paling relevan, yang kemudian diproses oleh model bahasa besar (LLM) Google Gemini untuk menghasilkan jawaban yang akurat, memiliki dasar hukum yang jelas, dan dilengkapi dengan citation trace.

1.2 Tujuan CLARA

Tujuan utama CLARA adalah mendemokratisasi akses terhadap analisis hukum kontrak bagi seluruh lapisan pelaku usaha di Indonesia. Secara spesifik, CLARA bertujuan untuk:

  1. Membantu UMKM memahami risiko hukum dalam kontrak bisnis tanpa harus menyewa konsultan hukum penuh waktu.
  2. Menyediakan analisis kontrak yang cepat, terstruktur, dan dapat dilacak sumber hukumnya.
  3. Menjembatani kesenjangan pengetahuan hukum antara pelaku usaha kecil dan perusahaan besar.
  4. Menyediakan mekanisme retrievasi yang transparan melalui Retrieval Trace, sehingga pengguna dapat melihat persis darimana suatu jawaban berasal.
  5. Membantu pembuatan dokumen legal dasar seperti surat somasi dan review kontrak kerja (PKWT).

1.3 Mengapa CLARA Dibuat

CLARA dibuat atas dasar keprihatinan terhadap kondisi hukum bisnis UMKM di Indonesia. Data dari Kementerian Koperasi dan UKM menunjukkan bahwa hanya 8% UMKM di Indonesia yang memiliki akses terhadap pendampingan hukum dalam menjalankan usaha mereka. Artinya, 92% UMKM menjalankan bisnis tanpa perlindungan hukum yang memadai. Akibatnya:

  • Banyak UMKM menandatangani kontrak yang tidak seimbang dengan pihak vendor atau distributor.
  • UMKM sering mengalami kerugian akibat wanprestasi mitra bisnis tanpa mengetahui langkah hukum yang dapat ditempuh.
  • Perekrutan karyawan kontrak sering dilakukan tanpa pemahaman yang benar tentang aturan PKWT.
  • Biaya konsultan hukum yang tinggi menjadi hambatan utama (rata-rata Rp500.000-Rp2.000.000 per jam konsultasi).

CLARA hadir sebagai solusi pertama di Indonesia yang menggabungkan hybrid retrieval dengan AI reasoning khusus untuk konteks hukum bisnis UMKM.

BAB 2 — PENJELASAN SISTEM

2.1 GRIS Engine

GRIS Engine (Grounded Retrieval Information System Engine) adalah inti dari sistem CLARA. GRIS Engine bertanggung jawab untuk melakukan retrievasi informasi dari knowledge base hukum Indonesia dan dokumen kontrak pengguna, kemudian menggabungkan hasil retrievasi tersebut untuk diberikan ke LLM sebagai konteks dalam menghasilkan jawaban.

GRIS Engine mengimplementasikan arsitektur multi-stage retrieval:

  1. Query Processing — pertanyaan pengguna diproses dan diklasifikasikan oleh Intent Router untuk menentukan jenis pertanyaan.
  2. Parallel Retrieval — tiga metode retrieval dijalankan secara paralel: dense, BM25, dan symbolic.
  3. Score Normalization — setiap hasil retrieval dinormalisasi menggunakan skor antara 0 dan 1.
  4. Fusion — Reciprocal Rank Fusion menggabungkan semua hasil menjadi satu ranking terpadu.
  5. Context Assembly — hasil retrieval terbaik dirangkai menjadi konteks yang akan dikirim ke LLM.
  6. Reasoning — LLM menghasilkan jawaban berdasarkan konteks, dengan self-consistency check.
  7. Trace Generation — seluruh proses dicatat dalam Retrieval Trace untuk transparansi penuh.

2.2 Dense Retrieval

Dense Retrieval adalah metode pencarian berbasis vektor (vector search) yang menangkap kemiripan makna antara pertanyaan pengguna dengan konten dalam knowledge base. Tidak seperti pencarian kata kunci tradisional, dense retrieval dapat menemukan informasi yang relevan meskipun pengguna menggunakan istilah yang berbeda dari istilah dalam dokumen.

Proses dense retrieval di CLARA:

  • Pertanyaan pengguna di-embedding menggunakan model Xenova/multilingual-e5-base menjadi vektor 768 dimensi.
  • Vektor pertanyaan digunakan untuk mencari node Article dalam Neo4j yang memiliki vektor embedding paling mirip (cosine similarity).
  • Index vektor (article_embedding_idx) memungkinkan pencarian ANN (Approximate Nearest Neighbor) yang cepat.
  • Hasil: pasal-pasal dan dokumen yang secara semantik relevan dengan pertanyaan.
Contoh Penggunaan Dense Retrieval: Pertanyaan: "Kalau vendor tidak menjalankan kewajibannya, apa risikonya?" Dense retrieval akan menemukan pasal tentang wanprestasi meskipun kata "wanprestasi" tidak disebut dalam pertanyaan, karena makna "tidak menjalankan kewajiban" secara semantik mirip dengan konsep wanprestasi.

2.3 BM25 Retrieval

BM25 (Best Matching 25) adalah algoritma pencarian kata kunci tradisional yang telah diadaptasi untuk konteks hukum Indonesia. BM25 unggul dalam menemukan dokumen yang mengandung istilah hukum spesifik yang disebut secara eksplisit oleh pengguna.

Proses BM25 retrieval di CLARA:

  • Pertanyaan di-tokenisasi dan di-stem untuk bahasa Indonesia.
  • Full-text index Neo4j (article_text_idx) digunakan untuk mencari kecocokan.
  • BM25 menghitung skor relevansi berdasarkan frekuensi kemunculan kata kunci.
  • Hasil: pasal-pasal yang mengandung istilah hukum yang disebut dalam pertanyaan.
Contoh Penggunaan BM25 Retrieval: Pertanyaan: "Jelaskan Pasal 1243 KUHPerdata tentang ganti rugi." BM25 akan mencari node yang mengandung kata "Pasal 1243", "KUHPerdata", dan "ganti rugi" secara eksplisit.

2.4 Symbolic Retrieval

Symbolic Retrieval adalah metode penelusuran knowledge graph yang memanfaatkan relasi antar konsep hukum dalam basis data Neo4j. Berbeda dengan dense retrieval (berbasis makna) dan BM25 (berbasis kata kunci), symbolic retrieval menggunakan relasi terstruktur antar entitas hukum untuk menemukan informasi.

Proses symbolic retrieval di CLARA:

  • Intent Router mengidentifikasi konsep hukum yang relevan dari pertanyaan pengguna (misalnya: "wanprestasi", "somasi", "PKWT").
  • Node LegalConcept yang sesuai ditemukan dalam knowledge graph.
  • Relasi seperti REQUIRES, HAS_CONDITION, RELATED_TO, dan CITES ditelusuri untuk menemukan informasi terkait.
  • Traversal dapat dilakukan hingga kedalaman 1-2 hop untuk menemukan informasi yang relevan.

2.6 Fusion dan Reciprocal Rank Fusion

Reciprocal Rank Fusion (RRF) adalah algoritma penggabungan ranking yang digunakan untuk mengkombinasikan hasil dari beberapa metode retrieval. RRF bekerja dengan prinsip: dokumen yang muncul di peringkat atas pada lebih dari satu metode retrieval akan mendapatkan skor gabungan yang lebih tinggi.

Rumus RRF: RRF(d) = Σ 1 / (k + r(d))

di mana r(d) adalah peringkat dokumen d dalam suatu metode retrieval, dan k adalah konstanta (di CLARA menggunakan k=60).

Metode RetrievalBobotKelebihan
Dense Retrieval0.5Menangkap kemiripan semantik, toleran terhadap perbedaan istilah
BM25 Retrieval0.3Akurat untuk kata kunci spesifik, cocok untuk pencarian pasal
Symbolic Retrieval0.2Menangkap relasi hukum terstruktur, memberikan konteks hubungan

Keunggulan RRF di CLARA:

  • Tidak memerlukan training data untuk menentukan bobot optimal.
  • Robust terhadap metode retrieval yang memiliki performa berbeda-beda.
  • Memberikan hasil yang lebih stabil dibandingkan menggunakan satu metode saja.

2.7 Embedding

Embedding adalah proses mengkonversi teks menjadi vektor numerik yang merepresentasikan makna semantik dari teks tersebut. CLARA menggunakan model Xenova/multilingual-e5-base yang dijalankan secara lokal melalui Transformers.js dengan runtime ONNX.

Spesifikasi embedding CLARA:

  • Model: Xenova/multilingual-e5-base
  • Dimensi vektor: 768
  • Bahasa: Multilingual (termasuk Bahasa Indonesia)
  • Jalankan: Sepenuhnya offline (lokal)
  • Format: E5 query/passage prefixing
Mengapa embedding lokal penting? Embedding lokal memastikan CLARA dapat berjalan tanpa ketergantungan pada API eksternal untuk proses retrievasi. Hanya proses reasoning (generasi jawaban) yang membutuhkan Gemini API. Ini mengurangi biaya operasional dan meningkatkan privasi data pengguna.

2.10 Retrieval Trace

Retrieval Trace adalah fitur unggulan CLARA yang menyediakan transparansi penuh terhadap proses retrievasi. Setiap jawaban yang dihasilkan CLARA dilengkapi dengan trace yang menunjukkan:

  • Dense Leg: Hasil dense retrieval dengan skor cosine similarity.
  • BM25 Leg: Hasil BM25 retrieval dengan skor keyword matching.
  • Symbolic Leg: Hasil symbolic retrieval dengan jalur graph traversal.
  • Fusion Scores: Skor akhir setelah RRF, menunjukkan peringkat gabungan.
  • Source Citations: Pasal dan dokumen yang digunakan sebagai sumber jawaban.
  • Confidence Score: Skor kepercayaan yang dihitung dari self-consistency loop.
  • Groundedness: Verifikasi bahwa setiap klaim dalam jawaban didukung oleh sumber.
Nilai Utama Retrieval Trace: Retrieval Trace membedakan CLARA dari chatbot AI biasa. Pengguna tidak perlu percaya begitu saja pada jawaban CLARA. Mereka dapat memverifikasi sendiri sumber hukum yang digunakan, memeriksa relevansi setiap sumber, dan memahami bagaimana CLARA sampai pada kesimpulan tertentu.

3.4 Latar Belakang Masalah

Kopi Senja tidak memiliki staf legal atau in-house counsel. Wayan, sebagai pemilik, mengelola seluruh aspek bisnis sendiri dari produksi, pemasaran, keuangan, hingga legal. Ketika menerima draft kontrak dari BaliCepat yang terdiri dari 12 halaman dengan 15 pasal, Wayan merasa kewalahan dan khawatir membuat kesalahan yang dapat merugikan usaha kecilnya di masa depan.

Kekhawatiran utama Wayan meliputi:

  1. Klausul keterlambatan pengiriman — vendor bisa terlambat tanpa mekanisme peringatan yang jelas.
  2. Klausul denda keterlambatan pembayaran yang memberatkan.

SKENARIO 9 — HYBRID RETRIEVAL: TIGA JALUR SEKALIGUS

Tujuan: Puncak demo teknis — CLARA mengaktifkan dense, BM25, dan symbolic bersamaan lalu menggabungkannya dengan RRF.

  2. Setelah jawaban, buka Retrieval Trace.
  3. Tunjukkan 3 leg: dense, bm25, symbolic semuanya aktif.
  4. Jelaskan tabel RRF — fusion score.
  5. Tekankan CLARA tidak bergantung pada satu metode.

Q: Apakah selalu 3 jalur aktif?
Tidak. Intent Router menentukan jalur mana yang relevan. Untuk pertanyaan sederhana, mungkin hanya 1-2 jalur. Untuk pertanyaan kompleks, 3 jalur aktif paralel.
Q: Bagaimana jika hasil 3 jalur bertentangan?
RRF beri bobot lebih pada informasi yang didukung banyak jalur. Self-consistency loop verifikasi jawaban sebelum output final.

8.3 Skor dan Ranking

Setiap metode retrieval menghasilkan skor dan ranking yang dinormalisasi:

  • Dense Score: Cosine similarity antara embedding pertanyaan dan embedding artikel (0.0 — 1.0). Skor tinggi berarti makna artikel sangat mirip dengan pertanyaan.
  • BM25 Score: Skor BM25 yang dinormalisasi (0.0 — 1.0). Skor tinggi berarti artikel mengandung banyak kata kunci dari pertanyaan.
  • Symbolic Score: Skor berdasarkan relevansi node dalam knowledge graph (0.0 — 1.0). Dipengaruhi oleh jumlah hop dan relevansi relasi.
  • RRF Score: Skor fusion yang dihitung dari ranking di setiap leg. Artikel yang muncul di peringkat atas di banyak leg mendapat skor RRF lebih tinggi.

8.4 Reciprocal Rank Fusion (RRF) Detail

RRF bekerja dengan formula: RRF(d) = 1 / (k + r(d)) untuk setiap leg retrieval, kemudian dijumlahkan.

Contoh perhitungan untuk Pasal 1243 KUHPerdata:

  • Dense rank: 1 → RRF_dense = 1 / (60 + 1) = 0.01639
  • BM25 rank: 1 → RRF_bm25 = 1 / (60 + 1) = 0.01639
  • Symbolic rank: 4 → RRF_symbolic = 1 / (60 + 4) = 0.01563
  • Total RRF = 0.01639 + 0.01639 + 0.01563 = 0.0484 (setelah normalisasi: 0.0328)

Pasal 1243 mendapat skor RRF tertinggi karena muncul di peringkat atas di semua 3 leg retrieval. Ini menunjukkan bahwa informasi yang didukung oleh banyak metode retrieval lebih diandalkan.

8.5 Confidence Score

CLARA menghitung confidence score melalui self-consistency loop:

  1. LLM menghasilkan 3 jawaban dengan temperature berbeda (low=0.1, medium=0.4, high=0.7).
  2. Setiap jawaban di-embedding dan dihitung cosine similarity antar pasang.
  3. Mean pairwise cosine similarity menjadi semantic agreement score.
  4. LLM memeriksa apakah setiap klaim dalam jawaban didukung oleh sumber retrieval — ini menjadi groundedness score.
  5. Rata-rata dari semantic agreement dan groundedness menjadi overall confidence.

8.6 Pipeline Status

Pipeline status menunjukkan waktu eksekusi dan status setiap tahap dalam proses retrievasi dan reasoning. Ini membantu pengguna memahami berapa lama setiap tahap berlangsung dan mengidentifikasi potensi bottleneck.

Mengapa Retrieval Trace Penting? Retrieval Trace adalah fitur yang membedakan CLARA dari chatbot AI biasa. Chatbot biasa memberikan jawaban "kotak hitam" — pengguna tidak tahu dari mana jawaban berasal. CLARA memberikan transparansi penuh sehingga pengguna dapat: (1) Memverifikasi sumber hukum yang digunakan, (2) Memahami bagaimana kesimpulan dicapai, (3) Menilai tingkat kepercayaan jawaban, dan (4) Mengambil keputusan berdasarkan informasi yang terverifikasi.

BAB 12 — HYBRID RETRIEVAL

Bab ini menjelaskan mengapa CLARA menggunakan tiga metode retrieval sekaligus (dense, BM25, symbolic) dan bagaimana ketiganya saling melengkapi untuk menghasilkan jawaban yang lebih akurat, komprehensif, dan terpercaya.

12.1 Mengapa Tiga Metode Retrieval?

Tidak ada satu metode retrieval yang sempurna untuk semua jenis pertanyaan hukum. Setiap metode memiliki kelebihan dan kelemahan:

12.3 Skenario Penggunaan Tiap Metode

Kapan Dense Retrieval Paling Berguna:

  • Pengguna bertanya dengan bahasa sehari-hari: "Gimana kalau vendor saya ingkar janji?"
  • Pengguna tidak tahu istilah hukum yang tepat: "Saya mau pegawai kontrak, aturannya gimana?"
  • Eksplorasi awal: "Apa saja risiko dalam kontrak distribusi?"

Kapan BM25 Retrieval Paling Berguna:

  • Pengguna tahu pasal spesifik: "Jelaskan Pasal 1243 KUHPerdata."
  • Verifikasi: "Apakah benar UU 13/2003 Pasal 59 mengatur PKWT?"
  • Pencarian presisi: "Cari pasal tentang wanprestasi di KUHPerdata."

Kapan Symbolic Retrieval Paling Berguna:

  • Pertanyaan prosedural: "Apakah perlu somasi sebelum gugat?"
  • Hubungan hukum: "Apa hubungan PKWT dengan masa percobaan?"
  • Analisis berjenjang: "Dari wanprestasi, somasi, ganti rugi, sampai gugatan."

12.4 Keunggulan Pendekatan Hybrid

  1. Cakupan lebih luas: Informasi yang tidak ditemukan oleh satu metode mungkin ditemukan oleh metode lain. Dense retrieval bisa menemukan konsep wanprestasi meskipun tidak disebut, sementara BM25 memastikan pasal yang disebut eksplisit tetap terbaca.
  2. Akurasi lebih tinggi: Ketika suatu informasi didukung oleh banyak metode retrieval, keyakinan terhadap kebenarannya meningkat. RRF memberikan bobot lebih pada informasi yang konsisten di berbagai metode.
  3. Robustness: Jika satu metode retrieval gagal (misalnya, dense retrieval menghasilkan false positive), metode lain masih bisa memberikan hasil yang benar. Ini membuat sistem lebih tangguh.
  4. Transparansi: Dengan melihat kontribusi masing-masing leg dalam Retrieval Trace, pengguna bisa memahami bagaimana jawaban disusun dan menilai keandalannya.
  5. Adaptif: Intent Router secara otomatis menyesuaikan jalur retrieval mana yang paling relevan untuk setiap pertanyaan, menghemat komputasi tanpa mengorbankan kualitas.
Contoh Konkret: Dalam skenario demo, prompt "Keterlambatan pengiriman yang menyebabkan kerugian, apakah ini wanprestasi menurut Pasal 1243, dan apakah perlu somasi?" secara sempurna menunjukkan kekuatan hybrid retrieval: - Dense menangkap makna "keterlambatan" dan "kerugian" sebagai wanprestasi. - BM25 menemukan "Pasal 1243 KUHPerdata" secara eksplisit. - Symbolic menelusuri relasi Wanprestasi → Somasi dalam knowledge graph. Ketiga hasil digabung dengan RRF untuk menghasilkan jawaban yang lengkap dan terpercaya.

14.2 Pertanyaan tentang GRIS Engine

11. Apa itu GRIS Engine?
GRIS Engine (Grounded Retrieval Information System Engine) adalah mesin retrievasi hybrid yang menjadi inti CLARA. Ia menggabungkan dense retrieval, BM25 retrieval, dan symbolic retrieval secara paralel, lalu menggabungkannya dengan Reciprocal Rank Fusion (RRF).
12. Apa perbedaan dense, BM25, dan symbolic retrieval?
Dense retrieval mencari berdasarkan kemiripan makna (semantic search), BM25 mencari berdasarkan kata kunci eksplisit (keyword search), dan symbolic retrieval menelusuri relasi dalam knowledge graph.
13. Mengapa menggunakan tiga metode retrieval?
Karena tidak ada satu metode yang sempurna. Dense unggul untuk makna, BM25 unggul untuk kata kunci, symbolic unggul untuk relasi hukum. Ketiganya saling melengkapi.
14. Apa itu Reciprocal Rank Fusion?
RRF adalah algoritma yang menggabungkan hasil dari beberapa ranking dengan formula 1/(k+r), di mana r adalah peringkat dan k adalah konstanta (60). Informasi yang muncul di peringkat atas di banyak metode mendapat skor gabungan lebih tinggi.
15. Apa knowledge graph CLARA?
Knowledge graph CLARA adalah basis data graf (Neo4j) yang menyimpan konsep hukum (Wanprestasi, Somasi, PKWT) dan relasi antar konsep (REQUIRES, HAS_CONDITION, REGULATED_BY) beserta pasal-pasal yang mengaturnya.
16. Siapa yang membangun knowledge graph?
Knowledge graph dibangun oleh tim legal CLARA yang terdiri dari ahli hukum Indonesia, diverifikasi melalui referensi silang dengan undang-undang dan peraturan yang berlaku.
17. Apakah knowledge graph bisa diperbarui?
Ya. Knowledge graph diperbarui secara berkala mengikuti perubahan peraturan perundang-undangan. Kami juga membuka kontribusi dari ahli hukum eksternal.
18. Berapa banyak data dalam knowledge graph?
Knowledge graph CLARA saat ini berisi lebih dari 2.000 node (pasal, konsep hukum, dokumen) dan 5.000+ relasi. Ini mencakup 6 undang-undang utama dengan ribuan pasal.
19. Bagaimana cara kerja embedding di CLARA?
CLARA menggunakan model Xenova/multilingual-e5-base yang mengkonversi teks menjadi vektor 768 dimensi. Model ini dijalankan secara lokal melalui Transformers.js dengan runtime ONNX, tanpa perlu API eksternal.
20. Apakah embedding membutuhkan koneksi internet?
Hanya untuk download model pertama kali. Setelah model ter-cache, embedding berjalan sepenuhnya offline.

14.3 Pertanyaan tentang Fitur

21. Fitur apa saja yang tersedia di CLARA?
CLARA memiliki 5 fitur utama: (1) Legal Q&A — tanya jawab hukum berdasarkan knowledge base, (2) Contract Review — analisis kontrak dengan guardrail check, (3) Document Drafter — pembuatan dokumen legal, (4) Retrieval Trace — transparansi sumber jawaban, (5) Export — download hasil dalam berbagai format.
22. Apakah CLARA bisa membaca file PDF?
Ya. CLARA dapat membaca PDF, DOCX, dan gambar scan (JPG, PNG) dengan teknologi OCR. Teks diekstrak dan dianalisis per-klausul.
23. Berapa maksimal halaman dokumen yang bisa dianalisis?
CLARA dapat menganalisis dokumen hingga 100 halaman dengan baik. Untuk dokumen yang lebih tebal, disarankan dibagi menjadi beberapa bagian.
24. Bisakah CLARA menganalisis kontrak dalam bahasa Inggris?
Ya, CLARA dapat membaca kontrak berbahasa Inggris. Namun, knowledge base hukum yang digunakan tetap hukum Indonesia, sehingga analisis mungkin kurang optimal untuk kontrak yang tunduk pada hukum asing.
25. Apakah CLARA bisa membuat kontrak dari awal?
Ya. Fitur Document Drafter memungkinkan pengguna membuat draft kontrak dengan mengisi data melalui formulir interaktif. CLARA akan menghasilkan draft kontrak yang sesuai dengan ketentuan hukum.
26. Bisakah CLARA membantu negosiasi kontrak?
CLARA dapat memberikan analisis risiko dan rekomendasi klausul yang perlu diperbaiki. Namun, proses negosiasi tetap dilakukan oleh manusia.
27. Apakah CLARA bisa mendeteksi klausul yang melanggar hukum?
Ya. Fitur guardrail check mendeteksi klausul berbahaya seperti denda berlebihan, pemutusan sepihak, waiver of rights, dan klausul yang melanggar UU Perlindungan Konsumen.
28. Bagaimana cara menggunakan Retrieval Trace?
Setelah CLARA menjawab pertanyaan, klik tombol "Lihat Trace" atau "Retrieval Trace". Panel akan menampilkan detail sumber, skor, dan proses fusion.
29. Apa perbedaan mode Chat, Review, dan Draft?
Chat — tanya jawab interaktif dengan konteks percakapan. Review — analisis mendalam dokumen kontrak. Draft — pembuatan dokumen legal baru.
30. Bisakah saya export jawaban CLARA?
Ya. CLARA mendukung export ke format PDF, DOCX, dan TXT. Export mencakup pertanyaan, jawaban, dan retrieval trace.

14.4 Pertanyaan tentang Teknis dan Infrastruktur

31. Teknologi apa yang digunakan CLARA?
Backend: TypeScript/Express, Neo4j (graph DB), Redis (queue), BullMQ. Frontend: React/Vite. AI: Google Gemini (reasoning), Transformers.js (embedding lokal).
32. Di mana data CLARA disimpan?
Data disimpan di server CLARA dengan enkripsi. Untuk deployment lokal, data disimpan di Neo4j lokal pengguna.
33. Apakah CLARA bisa di-deploy di server sendiri?
Ya. CLARA mendukung self-hosted deployment melalui Docker. Semua komponen (Neo4j, Redis, backend, frontend) dapat dijalankan dengan docker-compose.
34. Apakah CLARA bisa diintegrasikan dengan sistem lain?
Ya. CLARA menyediakan REST API yang dapat diintegrasikan dengan sistem lain. Dokumentasi API tersedia melalui Swagger di endpoint /api/docs.
35. Berapa response time CLARA?
Rata-rata 2-5 detik untuk pertanyaan sederhana, 5-15 detik untuk pertanyaan kompleks yang membutuhkan retrieval dan reasoning mendalam.
36. Apakah CLARA bisa offline?
Untuk mode offline penuh, diperlukan GPU yang cukup untuk menjalankan model LLM lokal. Saat ini, CLARA membutuhkan koneksi internet untuk Gemini API (reasoning). Retrieval (embedding) sudah berjalan offline.
37. Database apa yang digunakan CLARA?
Neo4j (graph database) untuk knowledge graph dan data aplikasi. Redis untuk antrian job dan cache.
38. Bagaimana dengan skalabilitas CLARA?
CLARA dirancang horizontal scaling. Neo4j dan Redis mendukung clustering. Backend dapat di-replica di beberapa instance.
39. Apakah CLARA menyediakan API publik?
Ya. REST API CLARA terdokumentasi melalui Swagger UI. API key dapat diperoleh melalui akun enterprise.
40. Bagaimana cara backup data CLARA?
Neo4j mendukung dump database. Redis mendukung RDB/AOF persistence. Disarankan backup rutin sesuai kebijakan perusahaan.
